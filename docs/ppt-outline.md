> **设计原则**：每页只传递 1 个核心信息，图表优先于文字，代码/截图占 50% 以上视觉面积

## 第 4 页：产品定位 —— 双时间维度体验

**标题**：产品定位：Day-1 Productive，Day-N Trustworthy
**核心信息**：

| 维度 | 承诺 | 体验 |
|---|---|---|
| T = 0（装上） | Day-1 Productive | 60 秒内"看过"历史，能回答高难度问题 |
| T > 0（持续） | Day-N Trustworthy | 新消息实时更新、冲突自动识别、旧记忆智能衰减 |

**视觉建议**：
- 时间轴设计：从左到右，T=0 到 T=N
- T=0 节点：闪电 icon + "/feishu-distill" 命令截图
- T=N 节点：循环 icon + feedback 闭环流程图

**演讲者备注**：> "我们不追求'越用越懂'的漫长学习期。我们的承诺是双重的：装上第一天就 productive，持续使用第 N 天依然 trustworthy——因为记忆会更新、会纠偏、会遗忘。"

## 第 5 页：核心创新 1 —— 第一人称记忆

**标题**：创新 1：第一人称记忆（`subject: 1st/2nd/3rd`）
**核心信息**：
- 市面上的方案做"企业知识库"，我们做"个人助手的上下文"
- 区分"我的任务"vs"指派给我的任务"vs"团队背景"

**视觉建议**：
- 三个记忆卡片并排：
  - 卡片 1（蓝色，1st）："我承诺周四前给 PM Lin 一版 PRD"
  - 卡片 2（橙色，2nd）："林老板指派给小陈：索引优化"
  - 卡片 3（灰色，3rd）："团队决策：缓存方案选用 Redis"
- 每个卡片标注 `subject` 标签和对应的五维标签

**演讲者备注**：> "这是我最想强调的创新点。OpenClaw 是一个个人持有的 Agent，它的记忆必须是第一人称的——'我答应的 deadline'、'我和小陈的分歧'、'我们团队的规约'。不是客观知识库，而是带主体视角的可执行上下文。"

---

## 第 6 页：核心创新 2 —— 五维标签体系

**标题**：创新 2：五维标签 × 五类记忆
**核心信息**：

| 维度 | 取值 | 解决什么问题 |
|---|---|---|
| Tense（时态） | past / present / future | "什么时候为真" |
| Source（来源） | passive / active_inject... | "怎么知道的" |
| Subject（主体） | 1st / 2nd / 3rd | "谁的事" |
| Structure（结构） | event / state / cycle / dependency | "该怎么用" |
| Abstraction（抽象） | fact / pattern / concept | "多通用" |

五类记忆：Task | Decision | Preference | Relationship | Lesson

**视觉建议**：
- 中心：五维雷达图（蜘蛛网图），一个记忆条目的标签可视化
- 右侧/底部：五个记忆类型 icon（任务勾选、决策分叉、偏好心形、关系网络、教训灯泡）
- 配色：每个维度一种颜色

**演讲者备注**：> "传统记忆系统按内容类型分类，这在企业协作中切错了维度。同一句'周五前交'既是任务又是对某人的承诺。我们的方案是给每个记忆同时打上五个维度的标签，形成交叉索引。"

## 第 7 页：核心创新 3 —— Prompt-as-Business-Logic v2

**标题**：创新 3：Prompt-as-Business-Logic v2
**核心信息**：
- 零规则引擎，全部业务逻辑封装在 skill prompt 中
- v2 采用 9 层架构，4 个跨行业 few-shot，带推理痕迹与自检机制
- v1 v2 并存，代码自动优先加载 v2

**视觉建议**：
- 左侧：9 层架构金字塔（从下往上）
  1. 角色与使命
  2. 记忆类型学
  3. 输出格式规范
  4. 决策框架
  5. 噪声分类学
  6. 边界案例处理
  7. 多场景 Few-shot
  8. 质量自检清单
  9. 输入处理
- 右侧：A/B 对比表格（v1 vs v2 关键特性覆盖）

**演讲者备注**：> "传统方案用 30+ Python 模块、5 张 SQL 表实现记忆治理。我们走完全相反的路：所有业务逻辑都是 prompt。评委打开 markdown 文件就能看懂系统怎么工作。v2 prompt 有 9 层架构、4 个跨行业 few-shot，还有强制推理痕迹和质量自检清单。"

## 第 8 页：技术架构与数据流

**标题**：技术架构：极简、可追溯、可扩展
**核心信息**：
- 数据流：飞书生态（lark-cli）→ Attention Filter → Distillation Prompt → L0 Active Memory → OpenClaw Agent
- 存储：纯 markdown + frontmatter（与 OpenClaw 原生 memory 同构）
- 召回：keyword + frontmatter 过滤，top-20 注入系统提示

**视觉建议**：
- 数据流图（横向 pipeline）：
  - 飞书 lark-cli → [过滤] → [LLM 蒸馏] → [L0 Markdown] → [Hook 注入] → OpenClaw
- 底部：三层存储示意
  - L0（亮）：< 2000 条 active markdown
  - L1（暗）：SQLite Archive（预留）
  - L2（虚线）：Evidence Pointer → 飞书 source of truth

**演讲者备注**：> "我们的架构哲学是'存储形态 = 产品哲学'。每条记忆都是一个可读的 markdown 文件，cat 一下就能理解。我们不做 RAG，不做向量数据库——因为企业协作记忆需要结构化关系、精确溯源和版本感，而这些是向量相似度给不了的。"

---

## 第 9 页：核心代码展示（3 个关键片段）

**标题**：核心代码：极简、鲁棒、可验证
**核心信息**：
1. **LLM 输出自校验**（4 重校验：必填字段、类型合法、证据溯源、时间线一致）
2. **Hook 注入**（top-20 active 记忆自动追加到系统提示）
3. **反馈闭环**（correct/outdated/noise/important 四种动作）

**视觉建议**：
- 三栏布局，每栏 15 行代码左右，高亮关键行：
  - 左栏：`validateMemoryFile` 函数（高亮 `VALID_TYPES`、`messageIds.has`、`createdAt < earliestEvidence`）
  - 中栏：`before_prompt_build` hook（高亮 `.filter(status === "active")`、`.sort(importance)`、`.slice(0, 20)`）
  - 右栏：`feishu-feedback` switch case（高亮四个 action 及其数值变化）
- 代码背景：深色主题，语法高亮

**演讲者备注**：> "代码层面我们追求极简和鲁棒。这是三个核心片段：四重自校验确保错误记忆不入库；hook 自动注入 top-20 记忆；feedback 命令让用户一句话就能纠正 AI。"

## 第 13 页：差异化对比 —— 我们为什么不一样

**标题**：差异化：不是 RAG，不是知识库，是个人长程上下文伴侣
**核心信息**：

| 对比项 | 通用 RAG | 企业记忆治理层（竞品） | **Mem-X** |
|---|---|---|---|
| 数据来源 | 手动上传文档 | 消息/文档自动采集 | **飞书全量历史 + 实时增量** |
| 记忆权属 | 客观知识库 | 企业资产 | **个人持有** |
| 召回方式 | 被动问答 | 候选审核 + 人工确认 | **主动注入 + 用户可控** |
| 时间感 | 无 | 版本链（无衰减） | **完整生命周期** |
| 第一人称 | 无 | 无 | **有** |
| 飞书集成 | 浅 | 中（未用 lark-cli） | **深（lark-cli 一等公民）** |

**视觉建议**：
- 三列对比表，Mem-X 列用品牌色高亮
- 顶部：一句话定位 —— "RAG 回答'你问我答'，我们回答'你没问但该说了'"

**演讲者备注**：> "最后说一下差异化。RAG 把文本打成 embedding，按语义相似度召回——但它区分不了'周五交'是 future 约束还是一般描述，也区分不了'我承诺'和'客观事实'。我们的方案用结构化 frontmatter 精确表达这些关系，用证据链精确溯源到单条消息。"

## 附录：PPT 制作技术建议

### 工具选择
- **推荐**：Beamer (LaTeX) 或 Marp (Markdown → PPT) 或 PowerPoint / Keynote
- **代码展示**：Carbon (carbon.now.sh) 生成语法高亮截图，或直接使用 IDE 截图
- **数据流图**：Draw.io / Excalidraw / Mermaid

### 配色方案
- 主色：深蓝 `#1a365d`（科技、可信）
- 辅色：亮青 `#38b2ac`（AI、创新）
- 强调：橙黄 `#ed8936`（关键数据、警示）
- 背景：白/浅灰为主，深色页用于过渡和强调

### 字体建议
- 标题：思源黑体 Bold / Inter Bold
- 正文：思源黑体 Regular / Inter Regular
- 代码：JetBrains Mono / Fira Code

### 截图清单（需提前准备）
1. `node demo-offline.mjs` 运行截图（终端全屏）
2. `/feishu-distill` 实际运行截图（如有真实 lark-cli 数据）
3. `/feishu-status` 输出截图
4. OpenClaw 对话截图（验证上下文注入）
5. `/feishu-feedback` 执行截图
6. 记忆文件 `cat` 内容截图（展示 frontmatter 格式）
7. 测试全部通过截图（`node run-tests.mjs` 等）
8. v1 vs v2 A/B 对比表格截图

### 录屏 Demo 建议
- 时长：控制在 90 秒以内
- 节奏：每个步骤 20-25 秒
- 配音：可配简洁旁白，或现场口述
- 备份：如果现场无法播放视频，用截图 + 口述替代
