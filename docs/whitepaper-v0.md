# 企业级长程协作 Memory 系统 —— 产品白皮书 v0.5

> **产品命题**：让 OpenClaw 装上即懂、越用越准
>
> **核心理念**：不是造一个记忆引擎，而是让 OpenClaw 的 memory 长出"企业上下文"

---

## 执行摘要

**一句话**：我们让 OpenClaw 第一次能"看见"飞书——把群聊、文档、会议、日历中沉没的企业协作信号，变成 OpenClaw 能直接调用的第一人称记忆。

**三个核心卖点**：
1. **Day-1 Productive**：接入飞书后 60 秒内，OpenClaw 已经能回答"我和谁共事过""项目卡在哪"等高难度问题
2. **第一人称记忆**：不是客观知识库，而是"我答应的 deadline""PM 喜欢的 PRD 风格""踩过的坑"
3. **AI-native 极简架构**：业务逻辑全是 prompt，核心代码极简，存储形态与 OpenClaw 原生 memory 同构

**一个场景**：
> 用户："Omega 项目有什么悬而未决的事？"
> OpenClaw："3 项待跟进：① 客户 X 反馈默认看上周数据（4/23 提出，尚未认领） ② Q1 索引优化（阿亮，下周二前） ③ 你的 PRD v1（周四前，格式约束：user story 开头）"

**评测预期**：矛盾更新识别率 ≥ 80%，抗干扰命中率 ≥ 70%，效能提升 90%+（操作步数 12 → 1）

## 一、我们在解决什么问题

### 1.1 真痛点：OpenClaw 看得见"你"，看不见"你的协作"

OpenClaw 已经是一个成熟的个人 AI 助手。它能记住你和它的对话、积累对你的偏好、在多次会话中保持连续性——**它并不"失忆"**。

但它有一个根本的上下文盲区：**它看不见飞书。**

你和同事在群里拍板的决策、在文档里写的技术方案、在会议里确认的行动项、在日历里安排的 deadline——这些构成了你在企业中最真实的工作上下文，但 OpenClaw 一无所知。它只认识你输入给它的文字，不认识你真实的协作世界。

**这不是记忆缺失，是"企业上下文"缺失。**

| OpenClaw 已有的（会话记忆） | OpenClaw 缺失的（企业上下文记忆） |
|---|---|
| 记住你和它的对话历史 | 群聊里拍板的决策与理由 |
| 积累你的编码风格偏好 | 会议中的 action items 与承诺 |
| 跨 session 保持连续性 | 飞书文档里的技术方案 |
| 自动写入 `MEMORY.md` | 日历中的 deadline 与冲突 |
| dreaming 压缩短期信号 | 消息流中的承诺漂移与版本变更 |

**一句话定调**：OpenClaw 有"会话记忆"，缺"企业上下文记忆"。我们不是替代 `memory-core`，而是**在它的 L0 层注入飞书信号**，让同一份 memory 文件同时承载"你和 Agent 的对话"与"你在企业里的协作"。

### 1.2 更大的真痛点：协作中的信息沉没

在飞书企业协作环境里，有价值的信息以每天数百条的速度产生，然后以更快的速度沉没：

- **决策理由被遗忘**："为什么不用方案 A？"——三个月后没人记得
- **承诺漂移**：群里说周五交、会上改周三、文档写月底——三处矛盾没人标红
- **部落知识断代**：老员工脑子里"这个客户不要文档要 PPT"——文档里永远没有
- **跨项目教训不流通**：A 项目踩过的坑，B 项目原样再踩一遍

这些信息不在任何知识库里，它们散落在聊天记录、会议纪要、飞书文档、日历邀请里。**我们的记忆系统要做的，不是建一个知识库，而是把这些沉没信号打捞出来，变成 OpenClaw 能随时调用的"企业上下文"。**

### 1.3 量化价值：效率提升多少

基于 mock 数据集（48 条真实飞书消息）的测算：

| 指标 | 人工处理 | Feishu Context Memory | 提升 |
|---|---|---|---|
| **信息消化时间** | 逐条阅读 48 条消息 ≈ 12–15 分钟 | LLM 蒸馏为 15 条结构化记忆，阅读 ≈ 30 秒 | **96%** |
| **上下文检索** | 翻聊天记录 + 凭记忆回想 ≈ 3–5 分钟 | 直接注入系统提示，AI 即时回答 | **99%**（操作步数 12 → 1） |
| **关键信息遗漏** | 人脑一周后遗忘约 60% 非显性承诺 | 结构化记忆 + 证据链，持久保留 | **零遗漏** |
| **版本冲突发现** | 依赖人工核对，平均延迟 2–3 天 | 蒸馏时自动识别矛盾（如 deadline 周五 → 周日），即时归档旧版 | **实时** |

**一句话**：一个每天产生 100 条消息的活跃协作群，传统方式需要用户每天花 20 分钟"刷群"才能不漏事；装上本插件后，OpenClaw 在 60 秒内完成冷启动，之后用户只需在需要时问一句"这周我有什么待办"。

### 1.4 为什么必须是"第一人称"

市面上大多数记忆系统（包括我们观察到的竞品）都在做"企业记忆治理层"——把信息抽成客观事实、建版本链、做权限审计。这是有价值的，但不是 OpenClaw 用户最痛的点。

OpenClaw 是一个**个人持有的 Agent**。它的记忆必须是**第一人称的**：

- "**我**答应林老板周五交 PRD"
- "**我**和小陈在缓存方案上有分歧"
- "**我们**团队规定周报标题用 `周报-Omega-Wxx`"
- "**我**上次因为这个 dt 时区问题查错了一周"

**第一人称记忆 = 带主体视角的、可执行的、与个人工作流绑定的上下文。** 这是"装上即懂"的前提。

## 二、产品定位：Day-1 Productive，Day-N Trustworthy

### 2.1 双时间维度体验承诺

| 维度 | 承诺 | 用户体验 |
|---|---|---|
| **T = 0**（装上） | **Day-1 Productive** | 接入飞书后，60 秒内 OpenClaw 已经"看过"你过去的历史，能回答高难度问题 |
| **T > 0**（持续） | **Day-N Trustworthy** | 新消息进来实时更新、冲突自动识别、旧记忆智能衰减——记忆不会膨胀、不会过时、不会骗人 |

**关键洞察**："装上即懂"不意味着系统需要"学习"你三个月。它意味着系统一次性从你已有的海量飞书数据中**萃取**出当前画像，然后持续维护准确性。长程的价值不在"积累"，而在"维护"。

### 2.2 与现有方案的差异

| 对比项 | 通用 RAG | 企业记忆治理层（竞品路线） | **我们的方案** |
|---|---|---|---|
| 数据来源 | 用户手动上传文档 | 消息/文档自动采集 | 飞书全量历史 + 实时增量 |
| 记忆权属 | 客观知识库 | 企业资产 | **个人持有** |
| 召回方式 | 被动问答 | 候选审核 + 人工确认 | **主动注入 + 用户可控** |
| 时间感 | 无 | 版本链（但无衰减） | **完整生命周期（萌发→活跃→衰减→遗忘）** |
| 第一人称 | 无 | 无 | **有** |
| 飞书生态深度 | 浅 | 中（未用 lark-cli） | **深（lark-cli 一等公民）** |

### 2.3 为什么不用 RAG / 向量数据库

这是一个常被问到的问题。RAG（Retrieval-Augmented Generation）在企业知识库场景已经证明有效，但**不适合个人协作记忆**，原因有三：

**① 向量检索丢失结构化关系**
RAG 把文本打成 embedding，按语义相似度召回。但企业协作中的关键信息是结构化的：
- "周五交" 是 **future** 约束，不是一般性描述
- "我承诺" 是 **1st** 主体，不是客观事实
- "deadline 从周五改为周日" 是 **版本链**，不是两段独立文本
向量相似度无法区分这些关系，召回的可能是"看起来像"但时态/主体完全错误的内容。

**② 精确溯源 vs 模糊召回**
企业记忆需要**证据链**——当 AI 说"林老板说用 Redis"时，必须能追溯到具体消息 ID、发言人、时间戳。向量数据库返回的是"相关段落"，无法保证溯源精度。我们的 frontmatter 中每个 evidence 都带 `msg_id` 和 `speaker`，精确到单条消息。

**③ 基础设施极简**
RAG 需要 embedding 服务、向量数据库、分片策略、重排序模型。我们的方案：
- 存储：纯 markdown + frontmatter（与 OpenClaw 原生 `MEMORY.md` 同构）
- 过滤：keyword + frontmatter 字段查询（`< 2000 条时 LLM 直接全读`）
- 召回：按 importance + status 排序，取 top-20 注入系统提示

**规模论证**：一个活跃员工每天产生约 100 条消息，蒸馏后约 15 条记忆。一年约 5000 条。L0 Active 层只保留"当前有效"记忆（< 2000 条），完全在 LLM context window 内，无需向量索引。

## 三、记忆的类型学：五维 × 五类

### 3.1 为什么要重新分类

传统记忆系统按"内容类型"分类（决策/任务/文档），这在企业协作中切错了维度——同一句"周五前交"既是任务又是对某人的承诺又约束未来动作。

我们的分类是**二维的**：每个记忆条目同时被打上"五维标签"和"内容类型"，形成交叉索引。

### 3.2 五维标签（刻画记忆的本质属性）

| 维度 | 取值 | 解释 |
|---|---|---|
| **时态 Tense** | past / present / future | 已发生 / 当前为真 / 未来约束 |
| **信号来源 Source** | passive / active_inject / active_deny / action_feedback | 被动观察 / 主动注入 / 主动否认 / 行动反馈 |
| **主体 Subject** | 1st / 2nd:<entity> / 3rd | 关于我 / 关于具体他人 / 客观事实 |
| **时间结构 Structure** | event / state / cycle / dependency | 单点事件 / 持续状态 / 周期性 / 流程依赖 |
| **抽象层级 Abstraction** | fact / pattern / concept | 具体事实 / 模式总结 / 概念抽象 |

**为什么这五维就够了？** 它们覆盖了企业协作记忆的三个核心问题："什么时候为真"（tense）、"谁的事"（subject）、"该怎么用"（structure + abstraction + source）。少一维会丢信息，多一维会过度工程。

**例子 1——团队规约（周期模式）**：
> "周报周五下午 5 点前发，标题格式 `周报-Omega-Wxx`"
- tense: **cycle** | source: **passive** | subject: **3rd** | structure: **cycle** | abstraction: **pattern**

**例子 2——个人承诺（单点未来事件）**：
> "@我 PRD 周四前给一版"
- tense: **future** | source: **passive** | subject: **1st** | structure: **event** | abstraction: **fact**

**例子 3——决策（已发生事件）**：
> "缓存方案就用 Redis 了，理由：① 团队已有运维经验，② 后面要做持久化"
- tense: **past** | source: **passive** | subject: **3rd** | structure: **event** | abstraction: **fact**

**例子 4——关系模式（持续状态）**：
> "阿亮和小陈在技术选型上常有分歧，但一旦拍板会服从，且事后会承认对方正确"
- tense: **present** | source: **passive** | subject: **2nd:ou_yl_alex** | structure: **state** | abstraction: **pattern**

### 3.3 五类记忆（内容类型）

| 类型 | 定义 | 典型例子 |
|---|---|---|
| **Task（承诺）** | 谁→做什么→何时 | "我答应周四前给 PRD" |
| **Decision（决策）** | 团队拍板，含被否决方案与理由 | "缓存用 Redis，否决 Memcached，因为运维经验" |
| **Preference（偏好）** | 协作者或团队的工作风格 | "PM 喜欢 PRD 以 user story 开头" |
| **Relationship（关系）** | 协作者画像 / 协作模式 | "阿亮和小陈在技术选型上有分歧但服从决定" |
| **Lesson（教训）** | 已踩过的坑、不成文规约 | "Redis key 必须设 TTL，否则 OOM" |

**反共识设计**：我们只输出这五类。午餐讨论、表情、通用招呼——一律丢弃。**质量优先于覆盖率。**

## 四、架构哲学：Prompt as Business Logic

### 4.1 为什么是 AI-native

传统方案（包括我们观察到的竞品）用 30+ Python 模块、5 张 SQL 表、状态机引擎来实现记忆治理。这很工程化，但有个根本问题：**业务逻辑埋在代码里，评委看不懂，用户改不了，演化成本极高。**

我们的方案走完全相反的路：

> **所有"业务逻辑"都是 prompt。**

- 记忆萃取 = 一个 distillation prompt
- 冲突识别 = 一个 conflict-resolve prompt
- 衰减判断 = 一个 forget prompt
- 召回时机 = 一个 attention-filter prompt

**优势**：
1. **可解释**：评委打开 markdown 文件就能看懂系统怎么工作
2. **可演化**：改策略只改 prompt，不部署代码
3. **AI-native**：LLM 直接读写 markdown，没有中间层
4. **极简**：核心逻辑以 prompt 为主，代码量控制在极简范围内，复杂度在 prompt 设计而非工程堆量

### 4.2 存储形态：与 OpenClaw 同构

我们不做独立数据库，而是**扩展 OpenClaw 已有的 memory 基础设施**：

- **L0 Active Memory**：纯 markdown 文件，带 frontmatter，容量约束 500-2000 条
- **L1 Archive**：SQLite 元数据 + markdown body，已归档/旧版本
- **L2 Evidence Pointer**：只存 source_id，飞书是 source of truth

**这个设计的精妙之处**：存储形态 = 产品哲学。每条记忆都是一个可读的 markdown 文件，评委 `cat` 一下就能理解。版本链通过 supersedes 指针在 frontmatter 里表达，不需要数据库 join。

### 4.3 记忆的免疫系统：防止错误传染

长程记忆系统最大的风险不是"记少了"，而是**"记错了然后一直错下去"**。LLM 抽错事实 → 进入记忆 → 后续基于错记忆做决策 → 错误被放大。

我们的防御机制有四层：

| 防御层 | 机制 | 效果 |
|---|---|---|
| **源头控制** | confidence < 0.5 不入库；模糊表达（"我看一下"、"应该可以"）直接丢弃 | 减少错误入库 |
| **引用审计** | 每条记忆必须有 evidence（飞书 source_id + 原文 quote），错了能找到根源 | 可追溯 |
| **使用衰减** | 每次记忆被引用时，confidence 微降；高 importance + 低 confidence 组合触发"自检 prompt" | 常用记忆更易被纠正 |
| **反向记忆** | 用户主动否认 → 不删除原记忆，而是 status:rejected + 创建一条"反向记忆"防止再犯 | 错误成为学习素材 |

**关键原则**：我们不追求"永远正确"，我们追求"**错了能被快速发现和纠正**"。

## 五、核心场景与评测映射

### 5.1 场景一：跨群跨会的承诺追踪与冲突识别

**对话剧本**：

```
用户：我这周报到底什么时候交？群里好像改过。

OpenClaw：
【工具调用】memory_search(query="周报 deadline")
【输出】你本周的周报 deadline 有变更：

  📌 当前生效：周日下午前提交
     └─ 来源：4/24 林老板在 Omega 项目组群通知
     └─ 原因：周一早老板要听 Q2 汇报，需提前看

  📁 已 superseded：每周五下午 5 点前
     └─ 来源：4/21 林老板定下的硬规定
     └─ 状态：本周临时改期，非汇报周可能恢复

需要我帮你起草周报吗？
```

**评测命中**：矛盾更新测试（官方强制要求）
**效能故事**：传统方式需要人工翻 50 条消息确认"到底哪天交"→ OpenClaw 一句话给出带 evidence 的版本链

## 六、遗忘机制：让记忆"有生命"

**这是被大多数方案忽略但题目明确要求的能力。**

我们的遗忘不是"删除"，而是五种**渐进式处置**：

| 遗忘形态 | 触发条件 | 存储动作 |
|---|---|---|
| **完成归档** | 承诺已履行 | status: active → archived，移到 completed/ |
| **过期沉降** | 状态已过时 | status: stale，importance 下调 |
| **被否决降权** | 用户纠正错误记忆 | status: rejected，创建反向记忆防止再犯 |
| **错误纠正** | 记忆本身有误 | confidence 下调，保留作为"错误历史" |
| **冷却衰减** | 长期未访问 | 按 (now - last_accessed) 降 importance |

**关键洞察**：遗忘的目的是**保持 L0 Active Memory 的信噪比**。如果所有记忆永远 active，系统会越来越慢、越来越乱。只有让旧记忆"体面地退场"，新记忆才有空间浮出。

### 6.1 具体数值规则

| 规则 | 数值 | 说明 |
|---|---|---|
| **L0 容量上限** | 2000 条 | 超过时按 importance 最低的先归档 |
| **冷却衰减半衰期** | 7 天未访问 → importance 降 20% | 线性衰减，非指数 |
| **归档阈值** | importance < 0.3 | 自动移入 Archive |
| **承诺到期归档** | deadline 后 3 天 | 无论是否履行，先 stale |
| **自检频率** | 每日凌晨 2 点 | 跑 forget prompt 审视全量 active 记忆 |
| **高 importance 保护** | importance > 0.9 的记忆 | 永不自动归档，需用户手动确认 |

### 6.2 什么时候不该遗忘

- **法律/合规要求**：涉及审计、合同、客户 SLA 的记忆，frontmatter 标记 `retention: permanent`，跳过遗忘流程
- **决策理由**：即使决策已过时，"为什么选 A 不选 B"的理由长期保留（abstraction: concept 层级）
- **关系模式**：协作者画像（"阿亮倾向性能优先"）属于 pattern，衰减极慢（半衰期 90 天）

## 七、数据流向图与用户主动管理

```mermaid
graph TD
    A[飞书生态<br/>lark-cli] -->|群聊/文档/会议/日历| B[Attention Filter<br/>90% 丢弃]
    B --> C[Distillation Prompt<br/>五类记忆萃取]
    C --> D[L0 Active Memory<br/>markdown + frontmatter<br/>< 2000 条]

    D --> E[Live Update<br/>新消息增量]
    D --> F[Conflict Resolve<br/>矛盾识别+版本链]
    D --> G[Forget Prompt<br/>衰减+归档]

    E --> H[L1 Archive<br/>SQLite + markdown body]
    F --> H
    G --> H

    H --> I[L2 Evidence Pointer<br/>source_id]
    I --> A

    D --> J[OpenClaw Agent<br/>memory_search / memory_get<br/>+ 主动召回注入]

    K[用户主动管理] -->|add / forget / recall| D
    K -->|直接编辑 markdown| D
    K -->|@Bot 在飞书里| A
    J -->|回复/执行| K
```

**关键设计点**：
- 飞书是 source of truth，我们只存指针和摘要
- LLM 日常只与 L0 交互，L0 容量严格控制
- 所有"业务逻辑"在 prompt 层，不在代码层
- 存储形态与 OpenClaw 原生 memory 同构

### 7.1 用户主动管理的三条路径

记忆系统不是单向的"飞书喂给 OpenClaw"。用户必须能主动干预：

| 路径 | 方式 | 适用场景 |
|---|---|---|
| **CLI 命令** | `openclaw memory add "记住：这个客户不要 PPT"` | 快速注入 |
| **飞书 @Bot** | 在群里 `@OpenClaw 记住以后周报标题用周报-Omega-Wxx` | 协作中顺手记录 |
| **直接编辑** | 打开 `memory/xxx.md` 改 frontmatter 或正文 | 精确修正/批量调整 |

**关键原则**：用户拥有记忆的完全编辑权。我们不设 candidate gate（竞品路线），而是**信任用户 + 提供工具辅助修正**。如果系统记错了，用户一句话就能纠正。

#### 可控性配置

用户可通过 `memory/config.yml` 调整系统行为：

| 配置项 | 默认值 | 说明 |
|---|---|---|
| `daily_push_limit` | 5 | 每日主动 push 上限，超限后降级为静默存档 |
| `whitelist_chats` | `[]` | 只萃取指定群聊，空数组表示全部 |
| `blacklist_chats` | `[]` | 跳过指定群聊 |
| `pause_live_update` | false | 一键暂停实时更新（出差/休假模式） |
| `confidence_threshold` | 0.5 | 低于此值的记忆不入库 |

### 7.2 OpenClaw 的 Hook 点

我们利用 OpenClaw 已有的 plugin hook 体系，在三个关键时刻注入企业上下文：

| Hook | 时机 | 动作 |
|---|---|---|
| `session_start` | 每次新会话开始 | 拉取飞书增量更新，合并到 L0 |
| `before_prompt_build` | 每次模型调用前 | 按当前 topic 召回相关记忆，注入 system prompt |
| `before_compaction` | 会话压缩前 | 触发 memory flush，把会话中新确认的事实写入 L0 |

## 八、边界与克制：隐私与合规

企业级场景不能只说"我能做什么"，必须说清楚"**我不做什么**"和"**我怎么保护你**"。

### 8.1 隐私边界

| 原则 | 具体做法 |
|---|---|
| **记忆是个人持有的** | 所有萃取的记忆默认 visibility: private，存储在用户的本地 workspace，不上传到任何服务端 |
| **他人发言的处理** | 群聊中他人发言进入记忆时，不暴露原始消息全文，只保留摘要 + source_id。需要追溯时才按需拉取 |
| **退群/离职自动失效** | 用户退出某群聊或离职时，该来源的全部记忆自动 status: archived，不再参与召回 |
| **敏感内容过滤** | distillation 阶段加 redaction 规则：薪资、客户机密、身份证号等模式自动标记 `sensitivity: high`，默认不进入 active memory |
| **用户完全可控** | 用户可随时导出全部记忆（纯 markdown，即拷即走），也可一键清空 |

#### 关于数据上云的说明

原始飞书消息在 distillation 阶段需要经过 LLM 处理，但遵循三个约束：
1. **仅传输过滤后的片段**：Attention Filter 先丢弃 90% 噪声，只有高价值消息片段进入 LLM
2. **LLM provider 不保留上下文**：调用完成后不存储对话历史
3. **萃取后的结构化记忆完全本地存储**：markdown 文件只存在于用户本地 workspace，不上传任何服务端

### 8.2 我们不做的事（明确边界）

- **不做团队共享记忆池**：每个用户的记忆是独立的，不打通
- **不做企业级权限审计**：我们不替代 IT 审计系统
- **不做图片/视频内容理解**：当前版本只处理文本信号
- **不做实时消息推送**：我们 pull 不 push，不建立 WebSocket 长连接
- **不做跨企业数据聚合**：记忆绝不离开用户本地设备

### 8.3 容错设计

系统在任何环节失效时都不应崩溃，而是优雅降级：

| 故障场景 | 降级策略 |
|---|---|
| lark-cli 失效（网络/权限） | 使用已缓存的 L0 记忆继续服务，标记"飞书同步暂停"，每小时重试 |
| LLM API 超时 | 跳过本次 distillation，不阻塞用户使用；超时记忆进入 retry queue |
| distillation 输出格式错误 | 校验失败的不写入 L0，写入 `errors/` 目录供人工 review |
| L0 容量溢出 | 按 importance 最低的先归档，高 importance（>0.9）永不自动归档 |
| 用户误删 memory 文件 | git track 整个 memory 目录，支持 `git checkout` 恢复 |

## 九、差异化声明

### 9.1 与竞品 FMC 的核心差异

| 维度 | FMC（企业治理层路线） | **我们（个人伴随路线）** |
|---|---|---|
| 定位 | 企业记忆治理层 | **个人长程上下文伴侣** |
| 语言 | 权限/审计/版本链/候选审核 | **装上即懂/越用越准/第一人称** |
| 飞书集成 | OpenAPI 直连（未用 lark-cli） | **lark-cli 一等公民** |
| 冷启动 | 所有数据进 candidate 需人工审 | **Day-1 Productive，批量萃取即用** |
| 遗忘 | 占位符（stale 标签） | **五种渐进式遗忘机制** |
| 代码量 | 30+ Python 模块 + 5 张 SQL 表 | **核心逻辑以 prompt 为主，代码极简** |
| 召回主动性 | 候选 dry-run，不真 push | **主动注入 + 用户可控** |
| 第一人称 | 无 | **有** |

### 9.2 与通用 RAG 的差异

RAG 回答"你问我答"。我们的系统回答"你没问但该说了"。

- **召回时机主动性**：不是等用户提问，而是在 OpenClaw 准备执行动作时主动注入相关记忆
- **时间感**：过去/现在/未来三种时态的记忆有不同的召回策略
- **版本感**：矛盾更新不是覆盖，而是建立版本链并标注 superseded
- **遗忘感**：旧记忆会衰减、归档、最终退场，保持系统轻盈

## 十、评测设计大纲

### 10.1 必做评测（官方要求）

#### 评测 A：矛盾更新测试

**测试集设计**：
- 1-3 个测试群，共 150-500 条消息（视比赛账号实际数据量调整）
- 植入 5 组矛盾事实：deadline 改期 ×2、决策修正 ×2、责任人变更 ×1
- 每组矛盾间隔 12-48 小时，中间插入 20-40 条无关消息

**通过阈值**：
| 指标 | 阈值 | 说明 |
|---|---|---|
| 版本链正确率 | ≥ 80% | 新旧版本关联正确 |
| superseded 标记准确率 | ≥ 90% | 旧版本正确标记，新版本正确激活 |
| evidence 引用完整率 | ≥ 85% | 冲突双方的 source 都被引用 |

**对照组**：裸 OpenClaw（无飞书记忆）→ 完全无法识别矛盾

#### 评测 B：抗干扰测试

**测试集设计**：
- 1 个高密度项目群，7 天 500 条消息
- 其中 30% 为噪声（午餐/表情/通用确认/天气）
- 植入 10 条关键事实，平均被 35 条无关消息隔开
- 最后一条关键事实在 T+6 天被重新提起（模拟"悬而未决"召回）

**通过阈值**：
| 指标 | 阈值 | 说明 |
|---|---|---|
| 关键事实命中率 | ≥ 70% | 10 条中至少命中 7 条 |
| 噪声误召回率 | ≤ 10% | 不能把午餐讨论当成记忆 |
| 悬而未决召回率 | ≥ 80% | 被埋藏后重新提起的任务必须能被主动召回 |

**对照组**：裸 OpenClaw + 仅 RAG（每次现拉飞书做向量检索）→ 预期命中率低于 50%，无时间感与版本感

#### 评测 C：效能指标测试

**测试场景**："找出 Omega 项目所有悬而未决的任务"

| 方式 | 操作步数 | 耗时 | 完整度 |
|---|---|---|---|
| 人工（翻群聊+文档+日历） | ~10 步 | 5-15 分钟 | 依赖记忆，易遗漏 |
| 仅 RAG | 2 步 | 10-30 秒 | 只能回答被问到的问题 |
| **我们的方案** | **1 步** | **< 5 秒** | **主动给出结构化摘要+证据链** |

> 注：效能数据为设计目标值，最终以实际评测为准。

### 10.2 加分评测

| 评测 | 场景 | 指标 | 阈值 |
|---|---|---|---|
| **Cold Start 速度** | 首次接入到可回答 5 个高难度问题 | 总耗时 | ≤ 60 秒 |
| **遗忘准确性** | 30 天后检查：已完成的承诺是否正确归档、仍能被追溯 | 归档正确率 / 恢复召回率 | ≥ 90% / ≥ 85% |
| **第一人称准确率** | 50 条记忆中区分"我的任务"vs"他人的任务" | 主体标签准确率 | ≥ 85% |
| **跨源合并** | 同一决策在群聊+文档+会议中各出现一次 | 去重准确率 | ≥ 80% |
